# Loop Coverage Metric v1 Scope and Contract

Issue: **#1458**

Loop Coverage Metric v1 defines how Ouroforge describes the fraction of a project or run whose trusted changes and artifacts were produced or accepted through the evidence loop: **Seed → Build → Observe → Verify → Journal → Evolve**. The metric is descriptive. It measures loop authorship and loop verification coverage; it does **not** measure game quality, fun, accessibility, commercial readiness, production readiness, Godot replacement, or user value.

This is a scope/contract document for Era E Milestone 20. It adds no executable behavior, no auto-fix, no auto-apply, no auto-merge, no self-approval, no reviewer bypass, and no trusted browser mutation.

## Goals

- Define provenance classes for trusted changes and artifacts.
- Define the loop-coverage evidence artifact shape and coverage verdict states.
- Define the regression rule used by future Rust/local computation.
- Keep Milestone 20 limited to per-artifact provenance class attribution and coverage computation.
- Defer the full intent-to-promotion provenance bundle and audit surface to Era E Milestone 25.
- Preserve #1 and #23 as open governance anchors.

## Non-goals

- No implementation of the Rust computation, CLI command, dashboard panel, Studio panel, or demo behavior in this issue.
- No claim that high loop coverage means a game is good, fun, accessible, production-ready, releasable, or commercially viable.
- No public release, deployment, hosted telemetry, cloud service, database, auth, signing, store upload, or native/mobile/console export behavior.
- No direct browser or Studio trusted writes. Browser/dashboard/Studio surfaces remain read-only for trusted state unless a later explicitly scoped Rust/local trusted API owns persistence.
- No overlap with Milestone 25's full end-to-end provenance bundle.

## Provenance classes

| Class | Meaning | Minimum evidence | Boundary |
| --- | --- | --- | --- |
| `loop-produced` | The trusted change or artifact was generated by the evidence loop from an accepted seed/task/plan and is linked to build, observe, verify, journal, and evolve evidence. | A run or artifact reference with loop stage links and a trusted Rust/local validation/verdict record. | It proves loop authorship/verification for the artifact, not quality or release readiness. |
| `loop-verified` | The trusted change or artifact was produced manually or externally, then accepted through the evidence loop by validation, verification, journal, and review evidence. | Manual source identifier plus trusted Rust/local validation/verdict and loop evidence links showing acceptance. | It counts as loop-covered because the loop verified it, not because the loop authored it. |
| `manual` | The trusted change or artifact is neither loop-produced nor loop-verified. | Source/change metadata is present but lacks the minimum loop evidence links. | Manual does not mean invalid; it only means not covered by this metric. |

Unknown, malformed, stale, or unsupported inputs must not be silently coerced into `manual`. They produce an explicit `insufficient-data` or `unsupported` verdict depending on the failure shape.

## Loop-coverage evidence artifact schema

Future implementation issues own the Rust structs and serializers. This contract fixes the additive artifact shape they must preserve:

```json
{
  "schemaVersion": "loop-coverage-metric-v1",
  "projectId": "collect-and-exit",
  "runId": "run-123",
  "baselineRef": "previous-run-or-golden-ref",
  "inputs": [
    {
      "artifactRef": "runs/run-123/evidence/verdict.json",
      "artifactKind": "verdict",
      "sourceRef": "repo-relative-or-run-relative-path",
      "provenanceClass": "loop-produced",
      "loopStageRefs": {
        "seed": "...",
        "build": "...",
        "observe": "...",
        "verify": "...",
        "journal": "...",
        "evolve": "..."
      },
      "trustedValidationRef": "runs/run-123/evidence/validation.json",
      "notes": "optional bounded diagnostic text"
    }
  ],
  "counts": {
    "loopProduced": 4,
    "loopVerified": 3,
    "manual": 1,
    "totalTrusted": 8
  },
  "fractions": {
    "loopProduced": 0.5,
    "loopVerified": 0.375,
    "manual": 0.125,
    "loopCovered": 0.875
  },
  "verdict": {
    "state": "computed",
    "baselineLoopCovered": 0.9,
    "currentLoopCovered": 0.875,
    "dropThreshold": 0.05,
    "reasons": []
  },
  "boundary": "Descriptive loop authorship/verification coverage only; not quality, fun, production, release, or Godot replacement evidence."
}
```

Schema rules:

- `schemaVersion` is exactly `loop-coverage-metric-v1`.
- `projectId` is required when the metric is project-scoped; `runId` is required when run-scoped.
- `inputs[]` is append-only evidence data. Missing or malformed refs are explicit diagnostics, not silent success.
- `provenanceClass` is one of `loop-produced`, `loop-verified`, or `manual`.
- `totalTrusted = loopProduced + loopVerified + manual` for successfully computed artifacts.
- `loopCovered = (loopProduced + loopVerified) / totalTrusted` when `totalTrusted > 0`.
- Fractions must be deterministic numeric values in `[0, 1]` and should be serialized with stable precision by the future Rust/local implementation.
- Coverage artifacts, generated traces, and run outputs remain untracked unless explicitly fixture-scoped.

## Coverage verdict states

| Verdict | When used | Required diagnostics |
| --- | --- | --- |
| `computed` | Inputs are supported, totals are valid, and no regression drop exceeds the configured threshold. | Current counts/fractions and baseline relation when a baseline exists. |
| `insufficient-data` | Inputs are supported in principle but required refs, counts, trusted validation, or loop stage links are missing/stale/malformed. | Missing/stale/malformed refs and the minimum evidence needed to compute. |
| `regressed` | Inputs are computable and the current `loopCovered` fraction drops below the baseline by more than `dropThreshold`. | Baseline ref, baseline fraction, current fraction, threshold, and drop amount. |
| `unsupported` | The project/run shape or artifact kind is outside this metric contract. | Unsupported kind/scope and the follow-up needed before computation can be trusted. |

## Regression rule

A run regresses when all of the following are true:

1. Current and baseline artifacts both have `computed` loop coverage.
2. Baseline is selected from the previous trusted run for the same project, or from a committed golden fixture when no previous run is available.
3. `baselineLoopCovered - currentLoopCovered > dropThreshold`.
4. The drop is not explained by explicit unsupported or insufficient-data diagnostics.

Default `dropThreshold` for follow-up implementation should be conservative and fixture-scoped until the project has enough history to justify a different threshold. The regression rule blocks a loop-coverage claim; it does not block manual development, merge authority, or release authority by itself.

## Rust/local ownership and UI boundaries

- Rust/local validation owns metric computation, artifact validation, verdict serialization, and any future CLI contract.
- Browser, dashboard, and Studio surfaces may display trusted loop-coverage artifacts but must remain read-only for trusted state.
- Studio may draft observations or requests only if a later explicitly scoped Rust/local trusted API owns persistence and validation.
- This contract grants no browser command bridge, arbitrary shell execution, source apply authority, auto-fix, auto-apply, auto-merge, or reviewer bypass.

## Milestone 20 and Milestone 25 boundary

Milestone 20 records only per-artifact provenance class and computes loop coverage. It answers: **what fraction of trusted artifacts were produced by or verified through the loop?**

Milestone 25 owns the full intent-to-promotion provenance bundle and audit surface. It may connect intent, plan, generation, review, promotion, audit, and replayability into a broader provenance chain. That full chain is explicitly out of scope for this Milestone 20 contract and must not be backfilled here.

## Backward compatibility and generated state

- New loop-coverage artifacts are additive only.
- Existing scenario, evaluator, runtime, verdict, evidence, dashboard, and source-like fixture contracts remain backward-compatible unless a later issue includes an explicit migration note.
- Generated runs, coverage artifacts, traces, temp files, screenshots, videos, browser state, and local dashboard exports remain untracked unless intentionally fixture-scoped.
- Existing manual workflows remain valid; the metric describes coverage, not legitimacy.
