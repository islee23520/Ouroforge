### Candidate Milestone 67: Executor Operator Cockpit and Read-Only Runbook UX v1

Goal: make the completed local Elixir/OTP Studio executor understandable and safe to operate through a read-only cockpit/runbook surface. This extends operator UX, not executor authority.

Target deliverables: a cockpit trust-boundary contract; read-only campaign status model; task DAG/progress read model; blocked-reason and runbook explanation surface; telemetry/utilization panel; golden-parity and manual-fallback panel; a minimal read-only cockpit demo; Scenario Coverage v59; and a roadmap/#1 governance refresh.

Published issue set: #2002–#2006, #2008–#2011. Each M67 issue carries the standard contract sections: Goal, Final Implementation Scope, Success Criteria, Verification Method, Guardrails, Explicit Non-Goals, Implementation Approach, PR Decomposition, Over-Engineering Checklist, Drift-Prevention Checklist, Language Boundary, Critical Risk Review, and Definition of Done.

GO scope: campaign/task progress, DAG state, budget/retry/backpressure diagnostics, read-only telemetry, parity status, evidence/ledger/verdict links, blocker explanations, copy-only fallback commands, and human-review handoff prompts.

DEFER scope: hosted dashboard, remote telemetry mirror, remote proposal queue, remote workers, distributed scheduling, and any server/database-backed orchestration.

NO-GO scope: browser command execution, direct artifact/ledger/evidence writes, trusted writes, apply/release/merge/deploy actions, executor-owned artifact truth, self-certification, reviewer bypass, and automation of human intent/taste/legal/release judgment.

Success criteria: the cockpit improves operator understanding while preserving the two-plane invariant: Elixir/OTP remains local control plane only, Rust remains the data plane and source of truth, trusted writes remain review/apply/trust-gradient gated, and #1/#23 remain open.
